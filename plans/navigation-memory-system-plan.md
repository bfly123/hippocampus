## Requirements Summary

### Problem Statement
Hippocampus already generates a strong global structural map (`structure-prompt.md`) but lacks two production-critical capabilities:
1) dynamic local context injection (repomap-like navigation)
2) persistent operational memory across project state and AI collaboration sessions.

### Scope
In scope:
- Navigation graph and ranking for symbol/module/file relevance
- Three memory classes: knowledge, facts, conversation
- Incremental build/update pipelines from index/snapshots/session logs
- MCP tool surface for retrieval, jump, dependency tracing, and memory CRUD
- Multi-AI orchestration pattern using shared context packs

Out of scope (phase 1):
- IDE-native deep integration plugins
- Full semantic code execution sandbox
- Cross-repo federated memory

### Success Criteria
- [ ] Given a task query + working files, system returns top relevant files/symbols with explainable ranking within 1.5s on medium repos.
- [ ] Memory retrieval returns accurate knowledge/fact/conversation snippets with source evidence and freshness metadata.
- [ ] MCP tools can serve navigation and memory requests reliably for Codex/Claude/Gemini/OpenCode workflows.

### Constraints
- Must integrate with existing `hippocampus-index.json` and snapshot flow
- Supports LLM summarization/embedding and vector DB (Chroma or Qdrant)
- Must support MCP dynamic querying
- Must remain token-budget aware for prompt assembly

### Assumptions
- Snapshot cadence remains stable (`hippo index` + auto snapshot)
- Existing tag/signature quality is sufficient for bootstrapping navigation
- MCP server is allowed to read `.hippocampus/` artifacts and session metadata
- Initial rollout targets single-repo, local-first operation

## Architecture

### Approach
Use a **dual-plane architecture**:
- **Navigation Plane (Working Memory)**: fast, query-time graph ranking for task-local context
- **Memory Plane (Long-Term + Episodic Memory)**: persistent normalized memory entries with hybrid lexical/vector retrieval

Both planes are anchored to snapshot IDs and index entities to ensure consistency and explainability.

### Key Components
- **Index Adapter**: normalizes `hippocampus-index.json` + snapshots into internal entities
- **Symbol Graph Builder**: builds file/module/symbol graph from signatures, tags, module assignment, and optional import edges
- **Navigation Ranker**: personalized PageRank + heuristics (focus files, mentioned symbols, recency)
- **Context Pack Renderer**: assembles L0-L3 snippets under token budget with role-aware filtering
- **Memory Ingestor**: produces knowledge/fact/conversation records from docs/config/sessions/diffs
- **Memory Store Layer**: metadata store + vector store abstraction (`chroma` or `qdrant`)
- **MCP Server**: exposes navigation and memory tools with typed schemas
- **Collaboration Recorder**: captures decision logs and memory updates from AI sessions

### Data Model (Core)

```json
{
  "navigation_graph": {
    "snapshot_id": "20260211T030253_259178Z",
    "nodes": [{"id": "file:src/x.py", "kind": "file", "module": "mod:query-api"}],
    "edges": [{"src": "file:src/x.py", "dst": "sym:foo", "type": "defines", "weight": 1.0}],
    "stats": {"node_count": 1240, "edge_count": 6210}
  }
}
```

```json
{
  "memory_record": {
    "id": "mem:fact:api_timeout_default",
    "type": "fact",
    "subtype": "config",
    "title": "Default API timeout",
    "content": "Default timeout is 30s in config.py",
    "source": {"path": "src/hippocampus/config.py", "snapshot_id": "20260211T030253_259178Z"},
    "evidence": ["line:42"],
    "tags": ["config", "api", "timeout"],
    "entity_refs": ["file:src/hippocampus/config.py"],
    "freshness": {"last_validated_at": "2026-02-11T12:00:00Z", "stale_after_days": 30},
    "confidence": 0.93,
    "embedding_model": "text-embedding-3-large"
  }
}
```

### Retrieval Strategy
Hybrid scoring for both navigation and memory:

`final_score = 0.45 * graph_rank + 0.25 * lexical + 0.20 * semantic + 0.10 * recency`

Role-aware guardrails:
- source files prioritized over tests/config/docs in code injection
- facts require deterministic evidence linkage
- conversation memory decays fastest unless reinforced

### Data Flow
1. `hippo index` updates index and snapshot artifacts
2. `hippo nav build` incrementally updates navigation graph cache
3. `hippo memory build` ingests/updates memory records + embeddings
4. `hippo mcp serve` answers tool queries for AI agents
5. task execution writes new conversation/decision memory and optional ADR

## Implementation Plan

### Step 1: Schema and Storage Baseline
- **Actions**: Define schema versioned docs (`navigation-graph.json`, `memory-records.jsonl`), storage abstraction (`MemoryStore`) and backend adapters (`chroma`, `qdrant`).
- **Deliverables**: Typed models, serialization, migration/versioning hooks.
- **Dependencies**: Existing index/snapshot schema and constants.

### Step 2: Navigation Engine MVP
- **Actions**: Build graph extraction from index/signatures; implement personalized ranking from query context (`files_in_focus`, `symbols`, `intent`).
- **Deliverables**: `hippo nav query` and context-pack builder (`L2/L3` budgeted injection).
- **Dependencies**: Step 1 schema, existing search/expand outputs.

### Step 3: Memory Ingestion Pipelines
- **Actions**: Add three ingestors:
  - knowledge (docs/ADR/design summaries)
  - facts (config/API/env/known-issues deterministic extraction)
  - conversation (session events, decisions, failed attempts, fixes)
- **Deliverables**: `hippo memory build`, `hippo memory query`, freshness validators.
- **Dependencies**: Step 1 storage abstraction.

### Step 4: MCP Tooling Layer
- **Actions**: Implement MCP server and tool contracts:
  - `hippo.navigate`
  - `hippo.symbol_refs`
  - `hippo.dependencies`
  - `hippo.memory_query`
  - `hippo.memory_upsert`
  - `hippo.context_pack`
- **Deliverables**: MCP tool handlers, schema validation, auth/limits.
- **Dependencies**: Step 2-3 core APIs.

### Step 5: Multi-AI Collaboration Integration
- **Actions**: Define coordinator workflow for Codex/Claude/Gemini/OpenCode using shared context pack + decision logging.
- **Deliverables**: orchestration spec + CLI wrappers (`ccb-*` integration path), memory writeback protocol.
- **Dependencies**: Step 4 MCP availability.

### Step 6: Evaluation and Hardening
- **Actions**: Build benchmark suite for retrieval precision, latency, and token efficiency; add regression tests around snapshot drift.
- **Deliverables**: baseline metrics, quality gates, rollout checklist.
- **Dependencies**: complete MVP stack.

## Query Interface (MCP Tools)

### `hippo.navigate`
Input: `{query, focus_files[], snapshot_ref, budget_tokens}`

Output: ranked files/symbols + explanation + context snippet candidates.

### `hippo.symbol_refs`
Input: `{symbol, snapshot_ref, depth}`

Output: definition, callers, callees, related files.

### `hippo.dependencies`
Input: `{entity_id, direction, max_hops}`

Output: dependency subgraph and critical paths.

### `hippo.memory_query`
Input: `{types[], query, entity_refs[], time_range, top_k}`

Output: hybrid-ranked memory entries with evidence and confidence.

### `hippo.memory_upsert`
Input: `{records[], source, mode}`

Output: accepted/rejected entries with validation reasons.

### `hippo.context_pack`
Input: `{task, focus_files[], provider, token_budget}`

Output: compact provider-ready context package (overview + navigation + memory).

## Technical Considerations

- Incremental-first: updates keyed by `snapshot_id` and file hashes; avoid full re-embed.
- Explainability: every memory/navigation hit includes source evidence and scoring breakdown.
- Freshness policy: facts and knowledge carry TTL; stale entries downranked or quarantined.
- Token discipline: context pack has per-role quotas (source/test/docs/memory).
- ID stability: maintain alias map for renamed modules/files across snapshots.
- Security: redact secrets and private paths before embedding or external tool response.

## Risk Management

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Noisy ranking from weak edges | High | Med | Weight tuning, gold query set, fallback lexical scoring |
| Memory hallucination drift | High | Med | Evidence-required writes, confidence thresholds, stale pruning |
| Snapshot schema drift | Med | Med | Versioned schemas + migration adapters |
| Latency spikes with large repos | Med | Med | Incremental indexing, caching, async embedding batches |
| Multi-AI inconsistency | Med | High | Shared `context_pack` contract and decision-log memory |

### Alternative Approaches Considered

1. **Vector-only memory (no graph ranking)**
   - Rejected: weak structural relevance for code navigation.

2. **Graph-only navigation (no semantic memory)**
   - Rejected: misses architectural rationale, decisions, and collaboration history.

3. **Single monolithic memory store**
   - Rejected: different validation/freshness needs for knowledge, facts, and conversation.
